## 1. `app.py`

## Purpose of this file

`app.py` is the entry point of the project.

When you run:

```bash
python3 app.py
```

this is the file that starts first.

## Why this file exists

This file handles:

- menu flow
- user input
- calling the service methods
- showing output in the terminal

It does **not** keep the full routing logic inside it.
That logic is kept in `complaint_routing/service.py`.

This separation is useful because:

- `app.py` stays readable
- service logic stays centralized
- display and logic are not mixed too much

## Functions in `app.py`

### `build_service()`

#### What it does

Creates a fresh `ComplaintRoutingService` object.

#### What it uses

- `DEFAULT_CATEGORY_MAP`
- `build_default_agents()`

#### Why it exists

Instead of writing the service creation code in many places, this function keeps it in one place.

It is used:

- once at startup
- again when the user chooses reset

### `show_events(service, events, auto_pause=True)`

#### What it does

Prints each event generated by the service.

After printing all events, it also prints the full system snapshot.

#### Input

- `service`: the current service object
- `events`: list of `Event` objects
- `auto_pause`: whether to pause briefly between steps

#### Why it exists

The service builds event data, but `app.py` decides how to display that event list.

This function is the bridge between logic and terminal output.

#### Small detail

It also picks box colors based on the event title.
For example:

- assigned events use green
- waiting events use yellow

### `run_guided_demo(service)`

#### What it does

Runs the ready-made demo complaints from `DEMO_COMPLAINTS`.

Then it resolves two agents so you can see reassignment happen.

#### Why it exists

This is useful for presentation.
You can show the project without typing custom input each time.

#### Flow inside this function

- print a note
- loop through `DEMO_COMPLAINTS`
- call `service.submit_complaint()` for each complaint
- show events
- resolve `A2`
- resolve `A1`

### `submit_custom_complaint(service)`

#### What it does

Takes a complaint from the user through terminal input.

#### Steps inside it

- shows allowed categories
- asks for text
- asks for category
- asks for priority
- validates priority is between 1 and 5
- calls `service.submit_complaint()`
- prints the result using `show_events()`

#### Why it exists

This is the main interactive feature of the app.

### `resolve_agent_complaint(service)`

#### What it does

Asks the user for an agent ID and calls `service.resolve_agent()`.

#### Why it exists

This allows the user to simulate complaint completion.
That is important because the next waiting complaint may get assigned immediately.

### `view_state(service)`

#### What it does

Prints the current system snapshot.

#### Why it exists

Sometimes you only want to see the current system state without adding or resolving anything.

### `main()`

#### What it does

This is the main loop of the app.

#### Flow inside `main()`

- build service
- print header
- print intro note
- show menu repeatedly
- run the selected option
- exit on `0`

#### Why `main()` matters

This is the top-level controller of the whole terminal app.

## 3. `complaint_routing/models.py`

## Purpose of this file

This file defines the main data objects used in the project.

## Why dataclasses were used

`@dataclass` makes object creation simpler.

Without dataclasses, you would need to write `__init__()` manually for each class.

That would add extra code without adding much learning value in this project.

## Classes in `models.py`

### `Complaint`

#### Fields

- `id`
- `text`
- `category`
- `priority`
- `department`
- `status`
- `assigned_agent_id`

#### Why these fields exist

- `id`: unique complaint identity
- `text`: actual complaint message
- `category`: raw complaint category
- `priority`: urgency level
- `department`: which department will handle it
- `status`: `waiting`, `assigned`, or `resolved`
- `assigned_agent_id`: tells which agent got the complaint

#### Why `status` defaults to `waiting`

Because every complaint is waiting at the moment it is created.
Assignment happens later.

### `Agent`

#### Fields

- `id`
- `department`
- `available`
- `current_complaint_id`

#### Why these fields exist

- `id`: unique agent identity
- `department`: which department the agent belongs to
- `available`: whether this agent can take a new complaint
- `current_complaint_id`: tracks the complaint the agent is handling

#### Why `available=True` by default

Because agents start as free in the default setup.

### `Event`

#### Fields

- `title`
- `lines`

#### Why this class exists

This class is used only for output steps.

Instead of printing directly inside the service every time, the service returns `Event` objects.
Then `app.py` prints them.

That makes the output easier to organize.

## 4. `complaint_routing/sample_data.py`

## Purpose of this file

This file keeps the default input data separate from the logic.

## Objects in this file

### `DEFAULT_CATEGORY_MAP`

This is the base category-to-department dictionary.

#### Why it exists

It gives the project default routing rules.

#### Example

- `water` -> `Water Department`
- `electricity` -> `Electricity Department`

### `DEMO_COMPLAINTS`

This is a list of sample complaints used by the guided demo.

#### Why it exists

It makes the project easy to present and test.

### `build_default_agents()`

#### What it does

Returns the default list of agents.

#### Why a function is used instead of a fixed list variable

Because every time you reset the system, you want fresh `Agent` objects.
If the same old objects were reused, their state might already be changed.

So creating new agent objects each time is the correct design.

## 5. `complaint_routing/service.py`

## Why this is the most important file

This file contains the full working logic of the project.

If someone asks where the complaint routing system really works, the answer is:

`complaint_routing/service.py`

## Class in this file

### `ComplaintRoutingService`

This class stores the whole system state and all main operations.

## `__init__(self, category_map, agents)`

### What it does

Initializes the whole service.

### Variables created here

- `self.category_map`
- `self.fallback_department`
- `self.complaints`
- `self.department_heaps`
- `self.agents_by_department`
- `self._complaint_sequence`
- `self._heap_sequence`

### Why each one is needed

#### `self.category_map`

Stores cleaned category-to-department mapping.

#### `self.fallback_department`

Stores the department used if category is unknown.

#### `self.complaints`

Stores every complaint object by complaint ID.

#### `self.department_heaps`

Stores one heap per department.
This is where waiting complaints live.

#### `self.agents_by_department`

Stores agents grouped by department.

#### `self._complaint_sequence`

Generates complaint numbers like `C1`, `C2`, `C3`.

#### `self._heap_sequence`

Keeps order stable when two complaints have the same priority.

### Important design choice here

Agents are grouped by department during initialization:

```python
for agent in agents:
    self.agents_by_department[agent.department].append(agent)
```

This makes later department-level lookup simpler.

## `submit_complaint(self, text, category, priority)`

### What it does

This is the main method for adding a complaint into the system.

### Full work done by this method

1. clean the category
2. find the department
3. generate complaint ID
4. create complaint object
5. store complaint in `self.complaints`
6. create display events for received and routing steps
7. push complaint into department heap
8. build queue view for display
9. call `_attempt_assignment()`
10. return complaint and events

### Why this method is important

This is the method where complaint creation and first routing actually happen.

## `resolve_department(self, category)`

### What it does

Returns the department for a given category.

### Why it exists separately

This keeps routing logic in one small method.
That makes the code easier to read and reuse.

## `resolve_agent(self, agent_id)`

### What it does

Marks the current complaint of that agent as resolved and frees the agent.
Then it tries to assign the next waiting complaint in that department.

### Main steps inside it

1. find the agent
2. handle invalid agent case
3. handle already-free agent case
4. get the agent's current complaint
5. mark complaint as `resolved`
6. free the agent
7. create resolution event
8. call `_attempt_assignment()` for the same department

### Why this method matters

Without this method, complaints could be assigned once, but the system would not show how new waiting complaints move forward after resolution.

## `find_agent(self, agent_id)`

### What it does

Searches through all department agent lists to find the matching ID.

### Why it exists

This keeps agent search logic in one place instead of rewriting it inside `resolve_agent()`.

## `get_department_queue_view(self, department)`

### What it does

Returns the complaints of a department in sorted priority order for display.

### Why it exists

The raw heap structure is good for processing, but not very clear for terminal output.

This method converts the internal heap storage into a clean list of full complaint objects.

### Important point

This method is mainly for display and snapshot generation.

## `get_snapshot(self)`

### What it does

Builds a dictionary that contains the current full system state.

### What is included in the snapshot

- overall flow text
- department queues
- agent states
- totals of complaints, waiting, assigned, resolved

### Why it exists

This lets the terminal app show the full system state in one place.

## `_attempt_assignment(self, department)`

### What it does

This is the method that decides whether a complaint gets assigned or keeps waiting.

### Main logic inside it

1. get the department queue
2. find an available agent
3. create availability-check event
4. if queue is empty, return no-assignment event
5. if agent is not available, return waiting event
6. otherwise pop the top heap complaint
7. update complaint and agent state
8. return assignment event

### Why this method is very important

This is where the routing system turns from storage into action.

If `submit_complaint()` is the entry method, `_attempt_assignment()` is the decision method.

## `_find_available_agent(self, department)`

### What it does

Finds and returns the first available agent in the given department.

### Why it exists

It keeps department agent lookup separate from assignment logic.

## `_format_queue_line(self, complaint)`

### What it does

Converts one complaint object into a simple text line for box display.

### Why it exists

This keeps the formatting logic in one small helper instead of repeating the same string-building code.

## 6. `complaint_routing/visuals.py`

## Purpose of this file

This file handles terminal formatting.

It does not decide routing or assignment.
It only decides how things are printed.

## Variables in this file

### `RESET`

ANSI reset code used after coloring text.

### `STYLE_MAP`

Dictionary that maps color names like `cyan` or `green` to terminal color codes.

## Functions in `visuals.py`

### `_use_color()`

#### What it does

Checks whether the terminal supports color output.

#### Why it exists

If the output is not a normal terminal, colored codes may look bad.
This function prevents unnecessary coloring in that case.

### `style(text, *names)`

#### What it does

Applies one or more styles to the given text.

#### Why it exists

It keeps color formatting in one place.

### `pause(enabled=True, seconds=0.9)`

#### What it does

Adds a small delay between events.

#### Why it exists

This makes the guided demo easier to follow step by step.

### `print_header()`

#### What it does

Prints the project title and the main complaint flow at startup.

#### Why it exists

It gives the app a clear starting screen.

### `print_box(title, lines, color="cyan", width=88)`

#### What it does

Prints a clean box with a title and multiple lines.

#### Why it exists

This is the main output helper used throughout the app.
Without it, the terminal output would look messy.

### `print_snapshot(snapshot)`

#### What it does

Prints:

- system totals
- department heaps
- agent states

#### Why it exists

This gives a full state view after each action.

### `print_menu()`

#### What it does

Prints the main terminal menu.

### `print_tip(lines)`

#### What it does

Prints a note box.

#### Why it exists

It is used for short explanations before demo steps or user input.

## 7. How the files connect together

## Startup flow

```text
app.py -> build_service() -> ComplaintRoutingService(...)
```

## Complaint submission flow

```text
app.py -> submit_custom_complaint() -> service.submit_complaint()
```

Inside `submit_complaint()`:

```text
resolve_department() -> heappush -> get_department_queue_view() -> _attempt_assignment()
```

## Resolution flow

```text
app.py -> resolve_agent_complaint() -> service.resolve_agent() -> _attempt_assignment()
```

## Display flow

```text
service returns Event objects -> app.py calls show_events() -> visuals.py prints boxes
```

## 9. Final understanding of the project

The project is simple on purpose.

It uses a small number of files and a small number of data structures, but each one has a clear role.

That is why the project is easy to explain:

- data objects are in one file
- sample input is in one file
- logic is in one file
- terminal display is in one file
- app flow is in one file

If you understand `service.py` fully, you understand the core of the whole project.
